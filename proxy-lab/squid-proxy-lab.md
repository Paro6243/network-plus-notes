# Lab: Explicit & Transparent Proxy with Squid (Parrot OS)

## Objective
Set up Squid as a forward proxy, observe traffic via Wireshark/access logs, implement domain-based ACL blocking, and explore transparent (intercept) proxy mode with iptables.

## Environment
- Parrot OS (host machine)
- Squid 6.13
- Firefox as test client

---

## Part 1: Explicit Proxy Setup

### Install & verify Squid
```bash
sudo apt install squid -y
sudo systemctl status squid
```

### Configure browser to use the proxy
- Firefox → Settings → Network Settings → Manual proxy configuration
- HTTP Proxy: `127.0.0.1`, Port: `3128`
- Enabled "Also use this proxy for HTTPS"

### Observing traffic
```bash
sudo tail -f /var/log/squid/access.log
```
Confirmed requests routing through Squid — visible as `TCP_MISS/200` for fresh fetches and `TCP_TUNNEL/200` for HTTPS CONNECT requests.

**Log format reference:**
```
[timestamp] [duration] [client_IP] [RESULT_CODE/STATUS] [bytes] [METHOD] [URL] - [ROUTE]
```

## Part 2: Domain Blocking with ACLs

Added to `squid.conf`:
```
acl blocked_sites dstdomain .facebook.com
http_access deny blocked_sites
http_access allow localhost
```

### Bug #1: Config line merge
A nano editing slip merged the new lines onto an existing `http_access allow localhost` line, producing a single garbled instruction. Squid failed to parse it (`FATAL: Bungled /etc/squid/squid.conf line 1604`) and refused to start.

**Fix:** Re-edited the block cleanly as three separate lines. Validated with:
```bash
sudo squid -k parse
```

### Bug #2: Rule order
Even after fixing the syntax, Facebook still loaded. Root cause: Squid evaluates `http_access` rules **top to bottom** and stops at the first match. `allow localhost` was placed *before* `deny blocked_sites`, so local traffic was allowed before the block rule was ever checked.

**Fix:** Reordered so the deny rule comes first:
```
acl blocked_sites dstdomain .facebook.com
http_access deny blocked_sites
http_access allow localhost
```

**Result confirmed in log:**
```
TCP_DENIED/403 CONNECT www.facebook.com:443 - HIER_NONE
```
Other sites (GitHub, etc.) continued to load normally, confirming the block was selective.

**Key lesson:** In ACL-based systems (Squid, iptables, firewalls), rule *order* is as important as the rules themselves — most engines stop at first match.

## Part 3: Transparent (Intercept) Proxy

### Config changes
```
http_port 3128
http_port 3129 intercept
visible_hostname parrot
```
(Kept the original `3128` port alongside the new intercept port — see Bug #3 below.)

### iptables redirect rule
```bash
sudo iptables -t nat -A OUTPUT -p tcp --dport 80 -j REDIRECT --to-port 3129
```
Verified with:
```bash
sudo iptables -t nat -L OUTPUT -n --line-numbers
```

### Bug #3: Internal icon URL failure
After switching to intercept-only mode (`http_port 3129 intercept` with no plain `3128` port), Squid failed to start:
```
FATAL: mimeLoadIcon: cannot parse internal URL: http://parrot/...
```
This happens because Squid needs a normal (non-intercept) port to serve its own internal resources (icons, error pages) to itself — an intercept-only port can't handle that self-referential traffic.

**Fix:** Kept both `http_port 3128` and `http_port 3129 intercept` active simultaneously.

### Limitation discovered: same-host interception
With the iptables rule and intercept port both active, requests to `neverssl.com` returned:
```
ERROR: The requested URL could not be retrieved — Access Denied
```
Squid's built-in `http_access deny to_localhost` loop-protection was triggering. Redirecting `OUTPUT` chain traffic *on the same machine Squid runs on* confuses Squid's view of the destination, making it look like a loopback request.

**Conclusion:** Transparent/intercept proxying is designed for traffic passing *through* a gateway/router from *other* devices — not for traffic originating on the Squid host itself. Properly testing this requires a second client machine or VM routed through this host as its gateway, which is planned for a future networked lab (VM + LAN + server setup).

## Takeaways
- Config syntax errors can silently break more than the block you're adding — validate with `squid -k parse` before every restart.
- Rule order matters more than rule content in most ACL/firewall systems.
- Intercept mode has real architectural requirements (a serving port for self-requests, and traffic that's actually routed *through* the proxy host, not generated by it).
- Left the iptables rule removed and squid.conf's intercept lines in place for reuse in the upcoming multi-VM lab.
